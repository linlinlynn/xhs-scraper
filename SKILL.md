# 小红书帖子采集与整理

按关键词搜索小红书帖子，逐篇采集正文、评论、图片内容，按发帖人类型归档，生成总结报告。

---

## Step 0: 参数收集

运行前必须使用 AskUserQuestion 向用户确认以下参数：

**必填：**
1. **主关键词** — 搜索的核心关键词（如"无限谷"）

**可选（AskUserQuestion）：**
2. **附加关键词** — 额外的搜索词，用于补充采集（可留空）
3. **采集篇数** — 要查看多少篇帖子（默认 10 篇）
4. **输出目录** — 文件保存到哪里（默认 `~/Desktop/xhs-采集`）
5. **采集内容**（多选，默认选项打勾）：
   - 帖子正文（默认开启）
   - 评论内容（默认开启）
   - 图片 OCR（默认关闭，小红书正文常在图片中，开启后会截屏并做 OCR）
   - 帖子链接列表（默认开启）
6. **是否需要原始抓取文本单独存档**（默认是）
7. **评论采集深度**（单选）：
   - 页面默认加载（默认）— 仅采集页面首次渲染展示的评论（约 10-20 条高赞评论 + 官方回复），速度快，适合快速调研
   - 深度滚动采集 — 在帖子详情页循环执行滚动加载更多评论，覆盖面更广但耗时显著增加
   - 全量采集 — 滚动直到评论区无法加载更多为止（⚠️ 慎选：高互动帖可能有数百条评论，耗时极长）
   - 手动输入 — 用户指定每篇帖子采集多少条评论
8. **原始文本清洗**（每次采集时用 AskUserQuestion 询问）：
   - `document.body.innerText` 提取的原始文本包含大量页面 UI 元素噪音
   - 清洗规则不预设固定模板，每次采集时询问用户要清洗哪些内容
   - 常见清洗对象包括：
     - 小红书 UI 元素（导航栏、筛选标签、搜索建议、评论输入框等）
     - 页脚法律信息（ICP 备案、营业执照等，注意页脚在原始提取中通常出现两次）
     - 公司信息（"行吟信息科技（上海）有限公司"等）
     - 底部互动数据纯数字行
     - "说点什么..."、"发送"、"取消"等按钮文本
   - 用户可选择性跳过清洗，保留原始文本

收集完参数后，向用户确认参数摘要，然后开始执行。

## Step 1: 搜索

### 1.1 打开搜索页

使用 `chrome-devtools` MCP 的 `navigate_page` 打开小红书搜索页：

```
https://www.xiaohongshu.com/search_result?keyword={主关键词}&source=web_explore_feed
```

URL 中的 keyword 需要做 URL encode（中文部分）。

### 1.2 前置检查

使用 `take_snapshot` 获取页面内容。检查快照中是否存在以下特征：

- 如果页面显示"登录""注册"等提示，或快照中无搜索结果，说明**未登录**。此时应：
  1. 提示用户先在浏览器中手动登录小红书
  2. 等待用户确认登录完成
  3. 重新执行 Step 1.1

### 1.3 提取帖子列表

从搜索结果的 snapshot 中提取帖子信息。每个帖子需要记录：

| 字段 | 说明 |
|------|------|
| 标题 | 帖子标题文本 |
| 作者 | 发帖人昵称 |
| 日期 | 发帖时间（如有） |
| 链接 | 帖子详情页链接 |
| 点赞数 | 赞数（如有） |
| 收藏数 | 收藏数（如有） |
| 评论数 | 评论数（如有） |

**注意：** 小红书搜索结果页可能需要滚动加载。如果当前 snapshot 中的帖子数量不足目标篇数，使用 `evaluate_script` 执行滚动操作：

```javascript
() => {
  window.scrollTo(0, document.body.scrollHeight);
  return 'scrolled';
}
```

然后再次 `take_snapshot` 获取新加载的内容。重复此过程直到获得足够的帖子，或已无更多内容加载。

如果搜索结果不足目标篇数，记录实际可用数量，继续后续流程。

---

## Step 2: 逐篇采集

按顺序对每个帖子执行采集。

**⚠️ 重要：随抓随存。** 每采集完一篇帖子，**立即**将原始文本追加写入 `{输出目录}/raw/{关键词}_raw.md`。不要等所有帖子采完再一次性写入——从上下文记忆中总结会失真，raw 文件是 ground truth，后续生成报告时必须对照 raw 确认细节。

### 2.1 打开帖子详情页

使用 `navigate_page` 打开帖子详情页 URL。

### 2.2 提取正文内容

使用 `evaluate_script` 提取页面文本：

```javascript
() => {
  return document.body.innerText.substring(0, 8000);
}
```

如果用户选择了采集评论内容，并且帖子详情页的快照中包含评论区，则评论会包含在 `innerText` 中。如果评论区需要滚动加载，使用滚动策略加载更多评论。

### 2.3 图片 OCR（可选）

仅当用户在 Step 0 中选择了"图片 OCR"时执行：

1. 使用 `take_snapshot` 找到帖子图片区域的元素 uid
2. 使用 `take_screenshot` 对图片区域截图，保存到 `{输出目录}/raw/{主关键词}_截图/` 目录
3. 使用视觉模型分析截图内容，提取图片中的文字

### 2.4 实时存储原始文本

每采集完一篇帖子，**立即**将原始文本追加写入 `{输出目录}/raw/{主关键词}_raw.md`，格式如下：

```markdown
---
序号: 1
标题: {帖子标题}
作者: {作者名}
日期: {日期，如无则留空}
链接: {帖子URL}
点赞: {数字}
收藏: {数字}
评论: {数字}
分类: 官方/媒体/用户（在 Step 3 填充，采集时可先留空）
---

{页面提取的原始文本}

---

```

**⚠️ 核心规则：**

1. **绝对不能改内容** — 禁止编辑、缩写、总结、筛选或删除任何原始文本内容。raw 文件的唯一价值是作为 ground truth 保留完整的原始采集数据。
2. **排版要结构化** — `evaluate_script` 返回的原始文本是一整坨没有结构的 innerText，直接写入人读不了、AI 解析也费劲。agent 必须对原始文本做结构化排版后再写入 raw 文件。

**排版规则（只动格式，不动内容）：**

- 将帖子正文和评论区分为两个明确的区域，用二级标题区分
- 主评论：每条评论一行，格式为 `作者 · 日期/地区（互动数）：评论正文`，把散落的元信息合并到一行
- 回复评论：用缩进（`>` blockquote）嵌套在对应的主评论下方，格式同上
- 评论之间用空行分隔
- "赞""回复""展开更多回复"等按钮文本忽略不写入（这些是 UI 元素，不是内容）
- 多篇帖子之间用 `---` 分隔

**判断主评论 vs 回复的方法：** innerText 中，主评论通常有日期/地区行 + 互动数字行 + 作者行 + 正文行；回复紧跟在主评论之后，通常缺少日期/地区行或互动数字行，只有作者 + 正文。

**结构化排版后的示例：**

```markdown
---
序号: 1
标题: xxx
作者: xxx
日期: xxx
链接: xxx
点赞: 100
收藏: 50
评论: 30
---

## 正文

{帖子正文原始文本，原样保留}

## 评论区

小蓮寺 · 03-17云南（赞48）：好帅好帅放大看这张脸简直绝了
> 底特律在逃仿生人（唔想返工版） · 03-16上海：4.10～真的很好玩，完全就是你聊什么，他后期就是什么样的
> 克莱斯特 · 03-16广东（赞1）：APP啥时候上架

Vallhalla. · 03-16广东（赞48，回复1）：真的就素呆呆萌萌的样子…

---
```

> **注意**：如果 agent 无法准确判断某段文本是主评论还是回复（innerText 没有明确的结构标记），优先保证内容完整，不做强制嵌套，把不确定的部分作为平级评论处理。

### 2.5 原始文本清洗（可选）

仅当用户在 Step 0 参数 8 中选择了清洗时执行。

清洗在结构化排版之前进行。根据用户确认的清洗规则，从 `evaluate_script` 返回的原始文本中移除噪音内容，再做结构化排版写入 raw 文件。

清洗的对象是页面 UI 噪音（导航栏、页脚法律信息、"说点什么..."按钮文本等），**不是帖子正文或评论内容**。清洗后写入 raw 的文本仍然是完整的帖子内容，只是去掉了与帖子无关的页面元素。

### 2.6 结构化数据记录

在内存中维护一个帖子列表，用于后续分类和报告生成。每条记录包含：

```javascript
{
  index: 1,
  title: "标题",
  author: "作者",
  date: "日期",
  url: "链接",
  likes: 0,
  favorites: 0,
  comments: 0,
  content: "正文内容摘要",
  topComments: ["精选评论1", "精选评论2"],
  category: "" // 在 Step 3 中填充
}
```

## Step 3: 分类与归档

采集完成后，根据帖子特征将所有帖子分为以下三类：

### 分类规则

1. **官方帖**
   - 判断标准：作者名与主关键词完全匹配，或作者名包含主关键词且为精确的品牌/产品名
   - 例如：主关键词是"无限谷"，作者"无限谷Sol007"归为官方帖

2. **媒体/自媒体帖**
   - 判断标准：作者名包含以下关键词之一：
     - "日报"、"晚报"、"观察"、"资讯"、"新闻"
     - "课代表"、"测评"、"推荐"、"种草"、"评测"
     - "研究所"、"实验室"、"编辑部"
     - "攻略"、"指南"、"清单"
   - 或者：帖子互动数据（点赞+收藏）显著高于同类帖子的中位数（超过 3 倍）

3. **用户帖**
   - 判断标准：不属于以上两类的所有帖子

### 输出分类结果

向用户展示分类统计：

```
采集完成，共 N 篇帖子：
- 官方帖：X 篇
- 媒体/自媒体帖：Y 篇
- 用户帖：Z 篇
```

## Step 4: 生成报告

在 `{输出目录}/{主关键词}_整理.md` 中生成结构化报告，参考 `references/report-template.md` 模板。

报告包含以下部分：

### 4.1 按类型归档

三个子章节，分别列出：
- **官方帖**：标题 + 作者 + 正文 + 精选评论（如有）
- **媒体/自媒体帖**：标题 + 作者 + 正文 + 精选评论（如有）
- **用户帖**：标题 + 作者 + 正文 + 精选评论（如有）

每个帖子条目的格式：

```markdown
### N. {标题}
- **作者**: {作者}
- **日期**: {日期}
- **链接**: {URL}
- **互动**: 👍 {赞} | ⭐ {收藏} | 💬 {评论}

{正文内容}

> **精选评论：**
> - {评论1}
> - {评论2}
```

### 4.2 帖子链接列表

如果用户在 Step 0 中选择了"帖子链接列表"，在报告末尾附上所有帖子的链接：

```markdown
## 帖子链接列表

| 序号 | 标题 | 作者 | 链接 |
|------|------|------|------|
| 1 | xxx | xxx | https://... |
```

### 4.3 总结部分

基于所有采集内容生成总结，包含：

1. **产品/主题概况**：根据帖子内容概括搜索关键词对应的产品或主题
2. **核心反馈**：用户的主要评价、反馈方向
3. **舆论关键词**：高频出现的正面/负面词汇
4. **值得关注的帖子**：互动量最高或内容最有参考价值的 3-5 篇帖子

## Step 5: 原始素材归档

确保以下文件已正确生成：

| 文件 | 说明 |
|------|------|
| `{输出目录}/raw/{主关键词}_raw.md` | 所有帖子的原始抓取文本（逐篇追加） |
| `{输出目录}/raw/{主关键词}_截图/` | 图片 OCR 模式下的截图（仅当开启图片 OCR 时） |
| `{输出目录}/{主关键词}_整理.md` | 结构化报告 |

如果用户在 Step 0 中选择不单独存档原始文本，则跳过 `raw/` 目录的创建。

## Step 6: 输出到飞书（可选）

将报告或原始素材写入飞书 Wiki，方便团队协作查阅。

### 前置条件

用户需要提前准备好 `lark-cli`，本 skill 不负责认证和配置。如果遇到权限问题，参考 `lark-shared` skill。

```bash
# 安装（如未安装）
npm install -g @larksuite/cli

# 初始化配置
lark-cli config init

# 登录授权
lark-cli auth login
```

### 获取目标知识空间 ID

```bash
lark-cli wiki spaces list --as user
```

从返回结果中找到目标 space_id。

### 创建新 Wiki 文档

```bash
lark-cli docs +create \
  --as user \
  --title "{文档标题}" \
  --wiki-space "{space_id}" \
  --markdown "$(cat 报告文件.md)"
```

创建成功后返回文档 URL。

### 追加内容到已有文档

1. 先获取文档当前内容：
```bash
lark-cli docs +fetch --as user --doc "https://my.feishu.cn/wiki/xxx"
```

2. 将新内容与已有内容拼接后更新：
```bash
lark-cli docs +update --as user --doc "https://my.feishu.cn/wiki/xxx" --markdown "$(cat 拼接后的内容.md)"
```

### 常用场景

- **先写独立文档再合并**：先将采集结果创建为新 Wiki 文档供用户 review，确认后再手动或通过 `+update` 合并到主文档
- **直接追加到主文档**：用 `+fetch` 获取主文档内容 → 拼接新章节 → `+update` 写回

## 错误处理

### 浏览器不可用
如果 `chrome-devtools` MCP 工具调用失败，提示用户：
> chrome-devtools 不可用。请检查：
> 1. Chrome 浏览器是否已打开
> 2. Chrome DevTools MCP 是否正在运行
> 3. 是否需要配置 MCP server

**如果以上都正常但仍无法连接，可能是 MCP 调试端口未正确注册。尝试手动修复：**

```bash
# 1. 关闭所有 Chrome 进程
killall -9 "Google Chrome"

# 2. 以调试模式启动 Chrome（使用独立的用户数据目录，避免影响日常 Chrome）
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir=/tmp/chrome_debug_profile

# 3. 手动创建 DevToolsActivePort 文件（路径中的第二行是 Chrome 的 WebSocket 调试地址，需根据实际情况填写）
echo "9222
/devtools/browser/xxx" > ~/Library/Application\ Support/Google/Chrome/DevToolsActivePort
```

> **注意**：步骤 2 使用了 `/tmp/chrome_debug_profile` 作为独立用户数据目录，这样不会影响你日常使用的 Chrome 配置和登录状态。但小红书登录态可能需要重新登录。

### 页面加载失败
如果某个帖子页面加载超时或出错，记录该帖子的 URL，跳过继续下一篇，在报告中标记为"未成功采集"。

### 登录过期
如果采集过程中遇到登录状态失效（页面重定向到登录页），暂停采集，提示用户重新登录后继续。

## 扩展性说明

本 skill 的工作流框架（参数收集 → 搜索 → 采集 → 分类 → 报告）是通用的。平台特定的逻辑（搜索 URL、页面结构、提取方式）封装在 Step 1-2 中。未来添加其他平台（如抖音、微博）只需在 `references/` 中新增对应平台的配置参考。
